① Mode = None — No mode selected at startup

Currently the app starts with DCO pre-selected and creates a tab immediately. This confuses users who want INI mode — they have to switch, causing an unintended extra tab. The fix: on startup (and every new tab), the session starts with no mode chosen. A centred placeholder tells the user exactly what to do. Select Folder and Recent are disabled/greyed until a mode is picked.

Key behaviour rules for Concept 1 • On app start: one tab, mode=None, placeholder shown, Select Folder + Recent disabled.
• Clicking + (new tab) always creates another mode=None tab with same placeholder.
• Once mode is chosen IN the placeholder, that tab transitions — no new tab opened.
• Tab title shows ⚙ until mode chosen, then 🔵/🟠 + folder name or "New Tab N".
② File Validation — Wrong file type blocked with clear popup

When a user in DCO mode accidentally selects an .ini file (via Select Folder or Recent), the app currently silently fails or loads nothing. The fix: a single validate_files_for_mode() function checks extensions before any decode call, shows a warning popup, and optionally offers to open the files in a new correct-mode tab.

③ Smarter Mode Button — Less confusing, context-aware

The current "Mode" button always opens a new tab when you switch, which users read as "I changed the current tab's mode". The fix: the button's behaviour depends on whether the active tab already has a mode set. It becomes a simple context indicator when data exists, and a chooser only for uninitialized tabs.

🔴 Current — Clicking mode always opens new tab, confusing
The confusion: User sees "DCO Mode" button and thinks "I'll switch to INI" → clicks → a new tab appears out of nowhere. They expected the current tab to change. Now they have two tabs.
